# Architecture: replayt-mcp-bridge

This document summarizes how the bridge is structured after the initial MCP tool surface is implemented in-process against replayt. It complements [MISSION.md](MISSION.md) (scope and trust boundaries) and [MCP_TOOLS.md](MCP_TOOLS.md) (tool catalog and schemas).

## Process and transport

- **Entry:** `python -m replayt_mcp_bridge` or the `replayt-mcp-bridge` console script → `replayt_mcp_bridge.__main__` → `server.run_stdio()` for the **stdio MCP server**, or → `health_probe.run_health_probe()` when the first argv token is **`health`** (same entrypoints)—a **short non-interactive** install probe that **exits** without entering the MCP loop—see [README.md](../README.md), [MISSION.md](MISSION.md#one-shot-operator-health-check-install-probe), and [Architecture review: one-shot operator health check (install probe)](#architecture-review-one-shot-operator-health-check-install-probe).
- **Transport:** [FastMCP](https://github.com/modelcontextprotocol/python-sdk) (`mcp.server.fastmcp`) with **`transport="stdio"`**. MCP clients speak JSON-RPC over the process stdin/stdout pair; the bridge does not open network listeners in this mode.
- **MCP server modules:** Tool handlers are grouped by domain across several modules under `src/replayt_mcp_bridge/`; **`server.py`** imports them for registration side effects and exposes **`run_stdio()`**. See [MCP server modules (maintainability split)](#mcp-server-modules-maintainability-split).

### MCP server modules (maintainability split)

**Trigger (documented threshold):** [MISSION.md](MISSION.md) originally kept a **single-file** server until the tool surface grew. Maintainers split the monolith when this backlog executed: the surface had **six** `@mcp.tool()` handlers and **`server.py`** was on the order of **~450 lines**—large enough that domain-sized PRs were awkward. **Future splits** are acceptable when a handler module crosses a similar line budget (rough guide **~300+ lines**) or the registered tool count jumps again.

**Import graph (stdlib / third-party omitted):**

```text
__main__.py ──► server.run_stdio()
                    │
                    ├── mcp_instance.mcp  (FastMCP singleton)
                    ├── observability     (configure_bridge_logging, emit_json_log)
                    ├── tools_health      ──► mcp_instance, tools_common
                    ├── tools_workflow    ──► mcp_instance, tools_common, replayt …
                    └── tools_persistence ──► mcp_instance, tools_common,
                                              persistence_support, observability, replayt …

tools_common  ──► observability.emit_json_log  (correlation + tool boundary logging)

persistence_support ──► replayt.cli.config (DEFAULT_LOG_DIR, resolve_log_dir),
                        replayt.persistence stores, observability (run-event allowlist default)
```

| Module | Role |
| ------ | ---- |
| `mcp_instance.py` | Shared **`FastMCP("replayt-mcp-bridge")`** instance used by every `@mcp.tool()` registration. |
| `tools_common.py` | **`_log_replayt_tool_boundaries`**, **`_tool_error`**, correlation **`ContextVar`** helpers; logger name stays **`replayt_mcp_bridge.server`** so structured stderr lines remain stable. |
| `tools_health.py` | **`replayt_echo`**, **`replayt_version_info`**. |
| `tools_workflow.py` | **`workflow_contract_snapshot`**, **`workflow_graph_mermaid`**, **`runner_dry_run_plan`**. |
| `persistence_support.py` | **`_split_typed_store_hint`**, **`_resolve_persistence_paths`**, **`_open_read_store`**, run-event projection helpers (no MCP registration). |
| `tools_persistence.py` | **`persistence_list_run_events`**. |
| `server.py` | Registers tools by importing **`tools_*`**, re-exports public callables and **`_split_typed_store_hint`** for tests, implements **`run_stdio()`**. |

## Layering (conceptual)

```text
MCP host (IDE, agent runtime, CLI wrapper)
        │ JSON-RPC / MCP tool calls
        ▼
FastMCP (schema from Python signatures, stdio framing)
        │
        ▼
replayt_mcp_bridge.server + tools_*  — validate/locate paths, map errors to JSON
        │
        ▼
replayt public APIs  — load_target, Workflow.contract, graph export,
                        validation_report, JSONLStore / SQLiteStore
```

**Rule:** Workflow semantics and persistence formats are **owned by replayt**. This repo owns **tool names**, **argument normalization**, **JSON-safe result shapes**, and **documented mapping** from each tool to a replayt surface.

## Tool groups

| Group | Tools | Role |
| ----- | ----- | ---- |
| Wiring / health | `replayt_echo`, `replayt_version_info` | Prove MCP wiring and report the resolved replayt version (integrator diagnostics). |
| Workflow introspection | `workflow_contract_snapshot`, `workflow_graph_mermaid` | Resolve a CLI-style **target** and expose contract and Mermaid graph text without running steps. |
| Runner (dry) | `runner_dry_run_plan` | Graph validation plus `validation_report` aligned with `replayt run --dry-check`; optional `inputs_json`, `strict_graph`, `metadata_json`, `experiment_json`, and `policy_hook_context_json` (see [MCP_TOOLS.md § Dry-check parity specification](MCP_TOOLS.md#dry-check-parity-specification-runner_dry_run_plan)). |
| Persistence read | `persistence_list_run_events` | Read-only access to events via JSONL log directory or SQLite path; default log dir matches CLI resolution when `store_hint` is omitted. Optional MCP **`event_fields`** and env **`REPLAYT_MCP_BRIDGE_RUN_EVENT_FIELDS`** (parsed in `observability.py`) may restrict returned dict-shaped events to listed **top-level** keys **before** any result copy—see [Architecture review: optional top-level event field allowlist](#architecture-review-optional-top-level-event-field-allowlist). Optional **`REPLAYT_MCP_BRIDGE_REDACT_RUN_EVENTS`** applies **`redact_structure`** to **`events`** afterward when truthy—default remains full pass-through for both steps. |

## Shared implementation patterns

- **Structured errors:** Operational failures that should reach the client as data use **`tools_common._tool_error(...)`** → `{ status: "error", tool, replayt_surface, message, correlation_id }` on mapped paths (see [MCP_TOOLS.md § Mapped failure paths](MCP_TOOLS.md#mapped-failure-paths-exception--branch-inventory)). `typer.BadParameter` from `load_target` and invalid run IDs are mapped this way instead of leaking stack traces across the MCP boundary. **`correlation_id`** matches structured stderr JSON for the same invocation per [MCP_TOOLS.md § Error response shape](MCP_TOOLS.md#error-response-shape).
- **Persistence resolution:** **`persistence_support._resolve_persistence_paths`** interprets `store_hint` (default dir, legacy path heuristics for JSONL directory vs `.sqlite`/`.db` file, or explicit `file:` / `jsonl-dir:` / `jsonl:` / `sqlite:` typed prefixes per [MCP_TOOLS.md § store_hint grammar](MCP_TOOLS.md#store_hint-grammar)). **`persistence_support._open_read_store`** yields a read-only store for `load_events`.
- **Schema stability:** Tool inputs are plain Python parameters on `@mcp.tool()` functions; hosts receive JSON Schema derived by FastMCP. Prefer additive optional parameters over breaking renames.

## Observability

- **Configuration:** `configure_bridge_logging()` (from `observability.py`) runs at server startup: stderr handler, default level **`INFO`**, overridable via **`REPLAYT_MCP_BRIDGE_LOG_LEVEL`**. All bridge-owned **`os.environ`** reads live in **`observability.py`** (log level, optional per-tool **`REPLAYT_MCP_BRIDGE_TOOL_TIMEOUT_<TOOL>_SECONDS`** and global **`REPLAYT_MCP_BRIDGE_TOOL_TIMEOUT_SECONDS`** for wrapped replayt-backed tools—built-in **300** s when those steps do not supply a positive limit per [MCP_TOOLS.md § Execution timeouts](MCP_TOOLS.md#execution-timeouts), optional **`REPLAYT_MCP_BRIDGE_STORE_HINT_ROOTS`**, optional **`REPLAYT_MCP_BRIDGE_RUN_EVENT_FIELDS`** default allowlist for `persistence_list_run_events`, and optional **`REPLAYT_MCP_BRIDGE_REDACT_RUN_EVENTS`**; see [SECURITY.md](SECURITY.md) and [Security review (phase 6)](#security-review-phase-6)).
- **Server lifecycle:** `run_stdio()` emits structured `replayt_mcp_bridge.server.start` with `transport: stdio` once before blocking on the MCP run loop.
- **Tools:** **`tools_common._log_replayt_tool_boundaries`** wraps every registered handler (including `replayt_echo`) and emits JSON lines for `replayt_mcp_bridge.tool.begin` / `.end` with **`tool`**, **`correlation_id`**, optional **`mcp_request_id`** from FastMCP `Context`, and result **`status`** on completion. Client argument values are never included. Extra structured fields are redacted via `redact_structure` before emission. Mapped **`_tool_error`** payloads use the same **`correlation_id`** (see [MCP_TOOLS.md § Error response shape](MCP_TOOLS.md#error-response-shape)).
- **Unhandled exceptions:** A structured `replayt_mcp_bridge.tool.unhandled_exception` line is logged, then `logger.exception` adds a traceback; the exception propagates and FastMCP / host behavior applies (see [MISSION.md](MISSION.md#security-and-trust-boundaries)).

## Non-goals (architecture)

- **Vendoring replayt** or reimplementing workflow execution here.
- **Implicit network or subprocess tool calls** beyond what replayt’s imported APIs already do when loading targets or stores.
- **Large generic “run arbitrary replayt CLI” tools** without explicit contracts—new tools should map to documented replayt capabilities like the mapping table in [MCP_TOOLS.md](MCP_TOOLS.md).

## CI and contributor automation

**Source of truth:** [.github/workflows/ci.yml](../.github/workflows/ci.yml) installs with `pip install -e ".[dev]"`, then runs **`ruff check`**, **`ruff format --check`**, and **`pytest -q`** as **separate steps** so the first failure is obvious. Pip cache uses `actions/setup-python` with `cache-dependency-path: pyproject.toml`. The workflow sets **`permissions: contents: read`** so the default `GITHUB_TOKEN` cannot write repository contents. The matrix covers Python **3.11**, **3.12**, and **3.13**; the **`replayt-floor`** job reinstalls **`replayt==0.4.25`** after the editable install to guard the declared lower bound in `pyproject.toml`.

**Documentation mirror:** [README.md](../README.md) (“Local checks”) and [CONTRIBUTING.md](../CONTRIBUTING.md) list the same Ruff and pytest invocations so contributors can reproduce CI without a shared script—duplication is intentional so each doc stands alone.

**Backlog alignment:** The “pytest + ruff CI + CONTRIBUTING expectations” item is structurally satisfied: workflow on PR/push (plus `mc/**` pushes), README and CONTRIBUTING document local commands and `pip install -e ".[dev]"` for Ruff, and [MISSION.md](MISSION.md#ci-and-contributor-automation) records the refined acceptance criteria. **Default branch green** remains an operational outcome after merge.
